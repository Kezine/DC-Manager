# Localisation (i18n) du client

Infrastructure de traduction de l'interface, adossée à **i18next**. Langues : **français**
(référence) et **anglais**. Cette page décrit le pattern à suivre pour toute chaîne d'UI.

> **État actuel.** Sont localisés : les **onglets** (domaine `tabs`), la page
> **Interventions** (`interventions`), et — depuis le **lot B1 « chrome & primitives »** —
> le **chrome** de l'app : primitives UI réutilisables (`ui`, dans `src-client/ui/*`),
> ossature/réglages/accueil (`shell`, `src-client/app/Shell.ts`), contrôleurs de documents
> fichier/REST et bootstrap (`app`, `src-client/app/{main,FileDocuments,RestDocuments}.ts`)
> et la vue Netmap (`graph`, `src-client/views/GraphView.ts`). Restent en français en dur
> (lots ultérieurs) : les **formulaires de vue** et les **vues d'administration**. La règle
> pour tout NOUVEAU code : **toute nouvelle chaîne UI passe par `I18n.t(...)`** (on ne
> rajoute plus de littéral français dans le rendu).
>
> **Hors périmètre (restent en français, par décision).** Les **messages d'exception**
> (`throw new Error(...)`, comparés parfois comme sentinelles — ex. `"permission-refusée"`),
> les **logs** (`console.*`, `Log.scope(...)`), les **commentaires** et les **noms par
> défaut persistés** (ex. `"Document 1"`). La **phase 2** transformera les erreurs serveur
> en **codes** traduits côté client (cf. plus bas).

## Pourquoi i18next

Choix de l'utilisateur (principe n°12 : proposer une librairie éprouvée plutôt que réinventer).
i18next est un moteur mûr, sans dépendance à un framework (compatible avec notre front TS
vanilla bundlé par webpack), qui gère nativement l'**interpolation** (`{{n}}`), le **repli de
langue** (`fallbackLng`) et le chargement de **catalogues en mémoire**. On l'enveloppe dans une
**classe sémantique à méthodes statiques** (`src-client/i18n/I18n.ts`, principe n°2) pour ne PAS
disséminer i18next dans tout le code : le reste de l'app n'appelle que `I18n.t`.

## Architecture

```
src-client/i18n/
  I18n.ts            # façade statique (init / t / locale / preference / setPreference)
  locales/
    fr.ts            # AGRÉGATEUR de RÉFÉRENCE (français) — importe les domaines, `export const fr = { tabs, … } as const`
    en.ts            # AGRÉGATEUR anglais (calque EXACT de fr)
    fr/              # un fichier PAR DOMAINE (`export const <domaine> = {…} as const`)
      tabs.ts  interventions.ts  ui.ts  shell.ts  app.ts  graph.ts
    en/              # calque anglais, mêmes fichiers
      tabs.ts  interventions.ts  ui.ts  shell.ts  app.ts  graph.ts
```

- **Un fichier PAR DOMAINE, un AGRÉGATEUR par langue.** À mesure que la migration ajoute des
  centaines de clés, chaque domaine vit dans `locales/<lang>/<domaine>.ts` (exporté `as const`) ;
  `locales/<lang>.ts` les **recompose** (`export const fr = { tabs, interventions, ui, shell,
  app, graph } as const`). Le **chemin d'import ne change pas** : `I18n.ts` importe toujours
  `./locales/fr` (et le test `dist-test/…/i18n/locales/fr.js`). Les nouveaux fichiers de domaine
  sont compilés **automatiquement** (le `include` de `tsconfig.node.json` couvre `src-client/**/*.ts`).
- **Catalogues en `.ts`, pas en `.json`.** Le `tsconfig` n'active pas `resolveJsonModule` ; un
  module `.ts` avec `as const` fige gratuitement le TYPE des clés et des valeurs (littéraux).
- **Clés structurées par DOMAINE.** `tabs` (onglets : `label` + éventuel `title` + `subtitle`),
  `interventions` (page Interventions), puis les domaines du **chrome** : `ui` (primitives
  réutilisables — inclut `ui.action.*` = Confirmer/Annuler/OK/Enregistrer/Supprimer/Fermer,
  réutilisés partout), `shell` (topbar/statut/réglages/accueil/en-têtes), `app` (documents
  fichier/REST, maintenance, boutons « + … » d'onglet, bootstrap) et `graph` (vue Netmap).
  Sous-structure lisible (`ui.modal.*`, `shell.settings.language`, `app.file.*`, `graph.frame.*`…).
- **Namespace i18next par défaut (`translation`).** Les ressources sont fournies EN LIGNE :
  `{ fr: { translation: fr }, en: { translation: en } }`. On accède donc aux clés directement
  (`I18n.t("tabs.equipements.label")`, sans préfixe de namespace).
- **`« Netmap »` reste `« Netmap »`** dans les deux langues : c'est un **nom de fonctionnalité**
  (l'onglet du graphe / carte réseau), pas une chaîne à traduire.

## Détection de la locale & préférence

`I18n.init()` résout la locale **effective** à partir de la **préférence persistée** dans
`localStorage["dcmanager.locale"]` :

| Préférence (`I18n.preference`) | Locale effective (`I18n.locale`)                         |
|--------------------------------|----------------------------------------------------------|
| `"fr"`                         | français                                                 |
| `"en"`                         | anglais                                                  |
| absente / inconnue → `"auto"`  | `navigator.language` commençant par `en` → anglais ; sinon **français (repli)** |

`init()` pose aussi `document.documentElement.lang` (accessibilité / moteurs). Elle utilise
`initAsync: false` pour que le chargement des catalogues (fournis en ligne) soit **synchrone** :
`I18n.t()` fonctionne dès le retour d'`init()`.

> **`init()` AVANT toute UI.** `I18n.init()` est appelé au tout début du bootstrap
> (`src-client/app/main.ts`, première ligne de `boot()`), **avant** la construction du `Shell`
> et des onglets. Un `I18n.t()` appelé avant `init()` **jette une erreur explicite** (garde-fou).

## Changer de langue : préférence + reload (choix v1 assumé)

`I18n.setPreference("fr" | "en" | "auto")` **persiste** la préférence puis **recharge la page**
(`location.reload()`), plutôt que de re-render l'app à chaud. Justification : l'interface est
construite **une seule fois** au boot ; un changement de langue est un événement **rarissime**.
Un rechargement complet est plus simple et plus sûr qu'un ré-affichage global (aucune vue à
réhydrater, aucun état — sélection, caméra 3D, modale ouverte — à recâbler). Le sélecteur de
langue vit dans **Réglages → « Langue / Language »** (`src-client/app/Shell.ts`).

## Ajouter / traduire une chaîne — les 3 endroits

Pour toute chaîne d'UI passée à la localisation, on touche **trois** points :

1. **`src-client/i18n/locales/fr/<domaine>.ts`** — ajouter la clé, avec le texte **français**
   (source de vérité). Un domaine NEUF : créer le fichier (`export const <domaine> = {…} as
   const`) et l'agréger dans `locales/fr.ts`.
2. **`src-client/i18n/locales/en/<domaine>.ts`** — ajouter la **MÊME** clé, avec la traduction
   anglaise (et l'agréger dans `locales/en.ts` si domaine neuf).
3. **Le site d'appel** — remplacer le littéral par `I18n.t("domaine.…")` (avec params
   d'interpolation optionnels : `I18n.t("x.y", { n: 3 })`).

Exemple (extrait du pilote onglets, `main.ts`) :

```ts
// fr.ts                              // en.ts
tabs: { racks: {                      tabs: { racks: {
  label: "Racks",                       label: "Racks",
  subtitle: "Baies : …",                subtitle: "Racks: …",
} }                                   } }

// main.ts
addListTab("racks", I18n.t("tabs.racks.label"), ListConfigs.racks, {
  subtitle: I18n.t("tabs.racks.subtitle"), …
});
```

## Garde-fou : test de complétude

`Tests/modules/test-i18n.js` (exécuté par `npm run test`) compare **récursivement** `fr` et `en`
et échoue si :

1. un **ensemble de clés diffère** (clé présente d'un côté, absente de l'autre — les écarts sont
   listés dans le message d'échec) ;
2. une **valeur est vide** ;
3. une **feuille n'est pas une chaîne**.

C'est ce qui empêche une traduction de « pourrir en silence » (clé oubliée à l'ajout, valeur
vidée par erreur). Toute nouvelle clé DOIT donc être ajoutée **des deux côtés**.

## Phase 2 prévue — messages serveur

Les chaînes produites par le **serveur** (erreurs d'API, libellés de validation) restent en
**français en dur** pour l'instant. La cible : le serveur renvoie des **codes** stables
(ex. `CertsError.code`) accompagnés de **paramètres**, et le **client** les traduit via `I18n.t`
(`errors.<code>` + interpolation). Cela garde le serveur agnostique de la langue de l'affichage
et centralise toutes les chaînes visibles dans les catalogues du client. Ce lot n'est PAS encore
implémenté ; à faire quand la couche d'erreurs serveur sera revue.
